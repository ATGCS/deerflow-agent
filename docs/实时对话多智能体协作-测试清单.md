## 1. 接口测试通用方法

### 1.1 创建任务（同时会创建一个 project bundle）

1. `POST /api/tasks`
2. 请求体：
   - `{ "name": "T1", "description": "..." , "thread_id": null }`
3. 返回体里会带：
   - `id`（task_id）
   - `parent_project_id`（project_id）

### 1.2 订阅项目 SSE

1. `GET /api/events/projects/{project_id}/stream`
2. 期望收到：
   - 命名事件：`event: connected`、`event: ping`
   - 业务事件：`data: <JSON>`，其中 JSON 结构包含：
     - `type`: 事件类型，如 `task:progress`
     - `data`: 事件 payload（字段按事件类型而定）

接口层验证建议用浏览器/脚本抓 `type/data` 即可。

### 1.3 注入业务事件（用于 SSE 验证与页面手动验证）

1. `POST /api/events/internal/broadcast`
2. 请求体（JSON）：
   - `{ "project_id": "<project_id>", "event_type": "task:progress", "data": { ... } }`
3. 同时带上：
   - `X-Internal-Events-Secret: ...`

前端 `collab-events.js` 约定：业务事件通过 SSE 默认 `onmessage`，并按 `payload.type` 与 `payload.data` 分发。

## 2. 功能点清单（逐条验证）

## A-01 — 门闩2 执行授权（任一放行）

### 验证接口：`/api/tasks/{task_id}/start` + `/api/tasks/{task_id}/authorize-execution`

1. `POST /api/tasks` 创建一个任务 `T_A01`（此时 `status=pending`，`execution_authorized=false`）
2. 立刻调用：`POST /api/tasks/{task_id}/authorize-execution`
   - 期望：HTTP `400`（或失败），原因应提到任务状态必须在 `planned/planning`
3. 调用：`POST /api/tasks/{task_id}/start`
   - 期望：返回 `Task started planning`，任务状态变为 `planning`
4. 调用：`POST /api/tasks/{task_id}/authorize-execution`，body 取：
   - `{ "authorized_by": "user" }`（或 `lead`）
   - 期望：`success=true`，响应内 `execution_authorized=true`
5. 再次调用第 4 步（幂等）：
   - 期望：仍 `success=true`，且不会报错

### 页面/端到端验证（建议作为补充）

- 最终 Worker 执行 gate 发生在 `task` 工具层（LangGraph/agent 跑起来时才真正“禁止执行”）。
- 如果你做端到端：在没 authorize 的情况下尝试触发协作执行，确认不会进入 worker。

## B-01 ~ B-04 — 数据模型落库（主任务、子任务 worker_profile、参数映射、统一建任务路径）

### B-01：主任务字段落库（`execution_authorized/thread_id` 等）

1. `POST /api/tasks` 创建任务
2. `GET /api/tasks/{task_id}`
   - 期望：返回包含以下字段（至少）：
     - `execution_authorized`（初始 `false`）
     - `thread_id`（你传什么就是什么；可以为 `null`）
     - `status`（初始 `pending`）
3. 调用 `POST /api/tasks/{task_id}/start`，再调用 `POST /api/tasks/{task_id}/authorize-execution`
4. 再次 `GET /api/tasks/{task_id}`
   - 期望：`execution_authorized=true`，`authorized_by` 与 body 一致

### B-02：子任务 `worker_profile` 落库

1. 在任务下调用：`POST /api/tasks/{task_id}/subtasks`
2. 请求体（示例）：
   ```json
   {
     "name": "S1",
     "description": "sub task",
     "dependencies": [],
     "worker_profile": {
       "base_subagent": "general-purpose",
       "tools": ["read_file"],
       "skills": [],
       "instruction": "Use tools safely.",
       "depends_on": []
     }
   }
   ```
3. 记录返回的 `subtask_id`
4. 调用：`GET /api/tasks/{task_id}/subtasks/{subtask_id}`
   - 期望：返回里存在 `worker_profile`，且关键字段（`base_subagent/tools/instruction`）一致

### B-03：`task` 工具参数映射（白名单/worker_instructions）

- 纯接口层很难直接验证“task 工具实际执行时的系统提示是否正确注入”。
- 建议使用（或保持）pytest：`test_task_tool_core_logic.py` 中的 worker_profile maps/tools/instruction 相关断言。

### B-04：统一建任务路径（REST 与 supervisor(create_task) 数据形状一致）

- 推荐接口层验证：只走 REST 创建，看返回结构是否与文档约定一致（`project.tasks[]`、`task.subtasks[]`、字段名统一）。

### H-02：SSE 客户端订阅是否正确解析 payload（type/data）并刷新事实

页面：`project-detail` 与 `task-detail`

#### 2.1 project-detail 的事实刷新

- project-detail 收到 `task_memory:updated` 后，会调用 `api.getProjectFacts(state.projectId)` 重拉事实并重渲染。
- 因此验证点：
  1. 在 project-detail 页面打开后
  2. 调用 `POST /api/task-memory/tasks/{task_id}/facts`
  3. 期望：事实列表在页面上新增对应 fact（或 facts 数量增加）

#### 2.2 task-detail 的事实刷新（需要重点验证接口是否存在）

- `task-detail.js` 在收到 `task_memory:updated` 时，会调用 `api.getTaskFacts(state.taskId)` 刷新 facts 列表。
- `tauri-api.js` 里 `getTaskFacts` 调用的是：`GET /api/task-memory/tasks/{taskId}/facts`
- 但你当前后端路由文件 `task_memory.py` 只实现了：
  - `POST /api/task-memory/tasks/{task_id}/facts`（写事实）
  - `GET /api/task-memory/tasks/{task_id}`（返回 TaskMemoryResponse，其中包含 `facts`）
- 所以这里你需要做“接口可用性验证”：
  1. 用浏览器/脚本执行：`GET /api/task-memory/tasks/{task_id}/facts`
     - 期望：如果返回 `404`，则 task-detail 的 facts 刷新会失败（页面事实可能不更新）
  2. 同时执行：`GET /api/task-memory/tasks/{task_id}`
     - 期望：返回里 `facts` 正常（如果你把 task-detail 的前端逻辑修到用这个接口，就能修复显示）

> 这一条是目前你“页面侧能否看到事实更新”的关键风险点，建议你把它作为 H-02 的优先复测项。

## 2.5 用户视角端到端：复杂任务生命周期演练（Planning → Assignment → Execution → Memory）

这段是按你说的“模拟用户角度”来写的：你在聊天里发起一个复杂任务，然后从“主智能体/子智能体/记忆/进度/分配信息”四个层面都能查到。

### 2.5.0 前置：准备一个可绑定的主任务

1. `POST /api/tasks`
2. body 例子：
   - `{ "name": "E2E-COLLK-1", "description": "e2e test", "thread_id": null }`
3. 从返回里记录：
   - `task_id`（主任务 id，记为 `MAIN_TID`）
   - `parent_project_id`（记为 `PROJ_ID`，用于 SSE 订阅）

### 2.5.1 用户发起复杂任务（聊天侧）

1. 打开 `#/chat`
2. 打开「复杂任务」开关（`chat-collab-mode`）
3. 在「协作任务 ID」输入框填入 `MAIN_TID`
4. 发送一条“明确可执行”的复杂目标（示例写法）：
   - “请把目标拆成可并行的步骤并执行。先对齐需求后再规划，然后我确认执行后继续。交付：……约束：……截止：……”
5. 观察聊天右侧栏/聊天窗口里：
   - “待确认（待你回复/选择）”是否出现
   - “任务进度（todos）”列表是否开始变化

> 这里的“门闩2”在页面侧通常体现为：直到模型进入正确阶段/用户回复后，协作执行才开始（你可以用没确认 vs 有确认做对照来验证 OR 放行）。

### 2.5.2 进度查询检查点（主智能体/任务状态）

在协作运行过程中，按以下顺序反复轮询（每 3~10 秒一次，直到完成）：

1. `GET /api/tasks`
   - 找到 `MAIN_TID` 对应条目（或直接用下一条）
2. `GET /api/tasks/{MAIN_TID}`
   - 验证“主智能体能获取到进度”的可观察证据（至少满足其一）：
     - `status` 从 `pending` → `planning/planned` →（最终）`executing`
     - `progress` 数值发生变化（从 0 到非 0 或持续更新）
     - 如果你开启了 SSE 页面：`project-detail.js` 的任务列同步更新（`task:started/task:progress` 事件触发时）

### 2.5.3 规划后分配检查点（主智能体规划→子智能体分配是否完成）

当 `GET /api/tasks/{MAIN_TID}` 中开始出现 `subtasks` 后，做以下校验：

1. `subtasks` 数量：应从空 → 非空（至少 2 个更符合“复杂任务”）
2. 每个 `subtasks[]` 元素：
   - `assigned_to`：应从 `null/未分配` → 变为某个 Agent（或你的 worker profile base 子智能体名）
   - `status`：应从 `pending`/`planning` →（最终会进入）`executing` / `completed/failed`
3. 子任务信息可查询：
   - `GET /api/tasks/{MAIN_TID}/subtasks/{SUB_TID}`
   - 期望返回的 `status/progress/assigned_to` 与上一步一致

### 2.5.4 分配后执行检查点（SSE/任务状态/子任务推进）

同时开一个 SSE 观察窗口（浏览器或 curl 抓流）：

1. `GET /api/events/projects/{PROJ_ID}/stream`
2. 观察事件类型：
   - `task:started`：data 中 `task_id` 应对应你子任务的 id（页面的 task-detail/subtasks 刷新就是靠这个）
   - 之后应出现：
     - `task:progress`（progress/current_step）
     - `task:completed` 或 `task:failed`
     - `task_memory:updated`（含 `facts_count`）

> 重要：当前实现里，TaskMemory 的“facts_count/事实写入”主要发生在子智能体完成之后（`persist_task_memory_after_subagent_run`），所以“处理中能不能查到记忆”更偏向：可以查到进度/状态；事实 facts 可能在完成后才稳定出现。你可以用这一点对照确认链路是否正确。

### 2.5.5 记忆查询检查点（子智能体执行过程/结束后能否查到）

对主任务与每个子任务（`MAIN_TID` 与每个 `SUB_TID`）做记忆查询：

1. `GET /api/task-memory/tasks/{MAIN_TID}`
   - 期望：返回包含 `facts`、`current_step`、`progress`、`output_summary`（以及 `status`）
2. `GET /api/task-memory/tasks/{SUB_TID}`（每个子任务都测）
   - 期望：
     - 子任务完成后 `facts.length > 0`
     - `progress` 反映完成进度（通常接近 100 或非 0）
     - `current_step` 与完成/失败对应

同时做“可用性对照”（验证前端 risk）：
3. （仅用于定位问题）`GET /api/task-memory/tasks/{SUB_TID}/facts`
   - 期望：当前后端可能返回 `404`（因为 `task_memory.py` 只有写 facts 与 GET `.../tasks/{task_id}`）
   - 如果 404：说明 task-detail 的 facts 刷新需要前端/后端对齐（你就可以用 `GET /api/task-memory/tasks/{task_id}` 修复显示）。

### 2.5.6 最终验收标准（你要的“主智能体/分配/执行/记忆/进度”都能查到）

满足任一（强制建议都做）：

1. `GET /api/tasks/{MAIN_TID}`：`subtasks` 非空，且每个子任务 `assigned_to` 可用
2. SSE 事件：每个子任务出现 `task:started`，且最终出现 `task:completed` 或 `task:failed`
3. 记忆 API：完成后 `GET /api/task-memory/tasks/{SUB_TID}` 返回 `facts` 非空；完成后 `GET /api/task-memory/tasks/{MAIN_TID}` 能反映汇总进展（`output_summary/current_step/progress`）

完成后，把本次 E2E 记为：
- `E2E-ComplexTask: 已完成（写清 MAIN_TID/PROJ_ID、出现的关键事件与 facts 数量）`
