# 多智能体协作与 Supervisor 系统设计

> 版本：v2.0（对话驱动任务创建）
> 日期：2026-04-02

---

## 1. 设计目标

构建一个**多智能体协作系统**，核心特点：

1. **对话驱动任务创建**：用户通过自然对话发起复杂任务，AI 主动询问并确认细节
2. **AI 自动拆解**：AI 判断任务复杂度，自动拆解为可执行的子任务
3. **Supervisor 监督者**：上帝视角，监督所有智能体进度，协调工作
4. **实时反馈**：在对话中实时汇报执行进度，用户可观察并干预

## 3. 核心角色

### 3.1 Supervisor（监督者）

**职责**：
- 任务分析和拆解
- 制定执行计划（Plan）
- 分配任务给子 Agent
- 监控执行进度
- 处理异常和阻塞
- 协调 Agent 之间的协作
- 汇总结果，向用户汇报

**特点**：
- 不直接执行任务（纯管理角色）
- 拥有全局视角
- 可以中断、重试、重新分配任务

### 3.2 Worker Agent（执行者）

**职责**：
- 执行被分配的具体任务
- 定期报告进度给 Supervisor
- 请求 Supervisor 协调资源或决策

**类型**：
- 通用 Agent（general_purpose）
- 专用 Agent（如代码Agent、搜索Agent、写作Agent）

## 4. 任务流程

### 4.1 任务生命周期

```
用户提交任务
       │
       ▼
┌──────────────────┐
│   任务登记        │  → 创建 Task 对象，状态：PENDING
└──────────────────┘
       │
       ▼
┌──────────────────┐
│   规划阶段        │  → Supervisor 分析任务
└──────────────────┘       │     拆解为子任务
       │                   │     确定依赖关系
       ▼                   ▼
┌──────────────────┐    ┌──────────────────────┐
│   计划确认        │───▶│ 等待用户确认计划      │
└──────────────────┘    └──────────────────────┘
       │                     │
       ▼                     ▼ (用户确认)
┌──────────────────┐    ┌──────────────────────┐
│   执行阶段        │◀───│ 开始执行             │
└──────────────────┘    └──────────────────────┘
       │
       ▼
┌──────────────────┐
│   完成/汇总       │  → Supervisor 汇总结果
└──────────────────┘       向用户报告
```

### 4.2 任务状态机

```
PENDING → PLANNING → PLANNED → EXECUTING → COMPLETED
   │           │          │           │
   │           │          │           ├──→ FAILED
   │           │          │
   │           │          └──→ PAUSED
   │           │
   └──→ CANCELLED
```

| 状态 | 说明 |
|------|------|
| PENDING | 任务已提交，等待处理 |
| PLANNING | 正在规划中 |
| PLANNED | 计划已生成，等待确认 |
| EXECUTING | 执行中 |
| PAUSED | 暂停（可恢复） |
| COMPLETED | 完成 |
| FAILED | 失败 |
| CANCELLED | 已取消 |

## 6. Supervisor 核心功能

### 6.1 任务规划器 (Task Planner)

**输入**：用户描述的任务
**输出**：可执行的子任务列表

**规划流程**：
1. 理解任务目标
2. 拆解为原子子任务
3. 确定执行顺序（考虑依赖）
4. 估算资源需求
5. 生成执行计划

**规划输出示例**：
```json
{
  "plan_id": "plan_xxx",
  "tasks": [
    {
      "id": "task_1",
      "name": "调研竞品信息",
      "description": "搜索并分析竞品功能",
      "agent_type": "researcher",
      "estimated_time": "10min",
      "dependencies": []
    },
    {
      "id": "task_2",
      "name": "撰写报告",
      "description": "基于调研结果撰写分析报告",
      "agent_type": "writer",
      "estimated_time": "15min",
      "dependencies": ["task_1"]
    }
  ]
}
```

### 6.2 进度跟踪器 (Progress Tracker)

**功能**：
- 实时跟踪每个子任务的状态
- 计算整体进度
- 检测阻塞或死锁
- 发送进度更新事件

### 6.3 协调管理器 (Coordination Manager)

**功能**：
- 任务分配和重分配
- 处理 Agent 之间的资源竞争
- 处理执行失败和重试
- 管理并行/串行执行逻辑

## 10. 上下文关联设计

### 10.1 三种上下文模式

```
┌─────────────────────────────────────────────────────────────────────┐
│                         模式一：完全共享                              │
│  Supervisor 与所有 Worker 共享同一上下文                               │
│  优点：Supervisor 完全了解所有情况                                     │
│  缺点：上下文膨胀严重，token 消耗大                                   │
└─────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────┐
│                         模式二：完全隔离（推荐）                        │
│  Supervisor 和每个 Worker 有独立上下文                                 │
│  通过「任务指令 + 结果摘要」传递信息                                    │
│  优点：高效、可扩展                                                   │
│  缺点：Supervisor 无法实时查看 Worker 内部状态                         │
└─────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────┐
│                         模式三：分层聚合（本章实现）                     │
│  Worker 定期向 Supervisor 报告进度（嵌入消息中）                        │
│  Supervisor 维护「全局状态摘要」                                        │
│  需要时可以深入查看某个 Worker 的完整上下文                             │
└─────────────────────────────────────────────────────────────────────┘
```

## 11. 增强监控能力（上帝视角）

### 11.1 Supervisor 可监控的内容

基于现有的 `SubagentExecutor`，Supervisor **可以**：

| 能力 | 支持情况 | 说明 |
|------|---------|------|
| 查看子任务最终结果 | ✅ 完全支持 | 通过 `SubagentResult.result` 获取 |
| 查看子任务执行日志 | ✅ 完全支持 | `SubagentResult.ai_messages` 包含完整对话 |
| 查看子任务执行状态 | ✅ 完全支持 | `SubagentResult.status` (PENDING/RUNNING/COMPLETED/FAILED) |
| 实时查看正在执行的内容 | ⚠️ 需要扩展 | 需要 SSE 推送或轮询 |
| 中断正在执行的子任务 | ❌ 需要扩展 | 需要新增 `interrupt` 功能 |

### 11.2 SubagentExecutor 返回结果结构

## 12. Agent 任务记忆机制（核心创新）

### 12.1 设计灵感

参考现有的**记忆模块**（Memory）实现，为每个子 Agent 维护专属的任务记忆。

**现有 Memory 模块的特点**：
- FileMemoryStorage：基于文件持久化，支持 per-agent 存储
- Facts 结构：关键事实以结构化方式存储
- API 接口：`/api/memory` 提供查询和修改

**Agent 任务记忆的创新点**：
- 子 Agent 每次执行后，提取并存储关键内容和事实
- Supervisor（上帝视角）可以查询任何子 Agent 的：
  1. 结果输出
  2. 关键事实
  3. 当前任务的特殊后台记忆
- 形成「共享记忆库」，供 Supervisor 做协调决策

### 12.3 任务记忆数据结构

```typescript
// 单个任务记忆
interface TaskMemory {
  task_id: string;
  agent_id: string;
  project_id: string;
  status: 'running' | 'completed' | 'failed';

  // 关键事实（由 Agent 提取）
  facts: TaskFact[];

  // 输出摘要
  output_summary: string;

  // 当前步骤描述
  current_step: string;

  // 进度
  progress: number;

  // 时间戳
  created_at: string;
  updated_at: string;
  completed_at?: string;
}

// 任务事实
interface TaskFact {
  id: string;
  content: string;           // 事实内容
  category: 'finding' | 'decision' | 'data' | 'conclusion';
  confidence: number;        // 0-1
  source_message?: string;   // 来源消息ID
}

// Agent 全部记忆
interface AgentMemory {
  agent_id: string;
  agent_name: string;
  tasks: TaskMemory[];       // 该 Agent 的所有任务记忆
  total_tasks: number;
  completed_tasks: number;
}

// 全局任务记忆汇总
interface GlobalTaskMemory {
  version: string;
  last_updated: string;
  agents: AgentMemory[];      // 所有 Agent 的记忆
  global_facts: TaskFact[];  // 跨 Agent 的全局事实
}
```

### 12.4 存储结构

```
.deer-flow/
├── memory.json                    # 全局记忆（已有）
└── task_memory/
    ├── project_{id}/
    │   ├── global_facts.json      # 项目全局事实
    │   └── agents/
    │       ├── agent_a/
    │       │   ├── task_1.json   # 任务1的记忆
    │       │   └── task_2.json   # 任务2的记忆
    │       └── agent_b/
    │           └── task_3.json
    └── index.json                # 索引文件
```

### 12.5 Supervisor 查询接口

```typescript

// 2. 获取某个任务的详细信息

// 3. 获取项目全局事实汇总

// 4. 获取所有 Agent 的运行时状态

```

### 12.6 与现有 Memory 模块的对比

| 特性 | 现有 Memory | Agent 任务记忆 |
|------|-------------|---------------|
| 作用域 | 全局用户记忆 | per-agent, per-task |
| 更新时机 | 对话后提取 | 任务执行后提取 |
| 主要用途 | 提供上下文 | 协调决策 |
| 数据类型 | 偏好、历史、背景 | 事实、发现、结论 |
| 访问权限 | Agent 读/写 | Supervisor 可读全部 |
| 生命周期 | 长期 | 与项目/任务同步 |

### 12.7 实现要点

1. **记忆提取**：在 SubagentExecutor 执行完成后，调用 LLM 提取关键事实
2. **持久化存储**：复用 FileMemoryStorage 机制，扩展支持 task_memory
3. **实时更新**：任务进行中定期更新进度和步骤描述
4. **查询优化**：建立索引，支持关键词搜索
