# 前端实时聊天与任务监控页面说明

**布局线框（可直接浏览器打开）**：[前端布局线框-聊天与任务监控.html](./前端布局线框-聊天与任务监控.html) — 含「实时聊天」「任务监控」两屏的栅格与区块示意。

本文回答三件事：**当前代码是否支持「先查询再创建子任务、并分配模板/技能/工具」**、**分配后的数据存哪里**、以及**实时聊天页与任务进度监控页应具备的形态**（目标体验与信息架构）。便于产品与研发对齐闭环。

---

## 一、代码侧是否支持该流程

**结论：支持。** 主智能体按文档「先 `list_subtasks` / `get_status`，再按需 `create_subtask` + `worker_profile_json`，再 `assign_subtask`」操作时，后端均有对应实现；执行子任务时 **`task` 工具**在传入协作 ID 时会从存储读取 `worker_profile` 并约束**子智能体模板、工具白名单、技能注入、额外 instruction**。

| 能力 | 实现要点 |
|------|----------|
| 先查询再建 | `supervisor(action=list_subtasks)` / `get_status` 返回子任务列表及 `worker_profile` 摘要（见 `supervisor_tool.py`） |
| 创建子任务并带技能/工具等 | `create_subtask` + `worker_profile_json`（JSON 字符串），字段对应 `WorkerProfile`：`base_subagent`、`tools`、`skills`、`instruction`、`depends_on` |
| 分配模板「智能体类型」 | `assign_subtask` 的 `assigned_agent` 写入子任务的 `assigned_to`（须为已配置的子智能体模板名） |
| 执行时生效 | `task` 传入 `collab_task_id` + `collab_subtask_id` 时，从存储加载该子任务的 `worker_profile`，与全局工具表求交后设置子智能体工具与系统提示（见 `task_tool.py`） |

**说明**：「先查询再创建」是**策略与提示词约束**；代码**不会**自动阻止未查询就 `create_subtask`。若需要硬约束，需在工具层或中间件增加校验（当前未做）。

## 二、分配与配置数据存哪里

### 1. 项目与任务计划（含子任务、`worker_profile`、`assigned_to`）

- **目录**（默认）：`{应用数据根目录}/.deer-flow/projects/`  
  - `base_dir` 来自 `deerflow.config.paths.get_paths()`，随部署变化。  
- **文件**：`project_{project_id}.json`（每个项目一个 JSON 文件）。  
- **结构要点**：  
  - `tasks[]`：主任务列表。  
  - 每个主任务下有 `subtasks[]`。  
  - 每个子任务可包含：  
    - `assigned_to`：`assign_subtask` 写入的**模板名**（如 `general-purpose`）。  
    - `worker_profile`：`create_subtask` 时由 `worker_profile_json` 解析并持久化的对象（`base_subagent`、`tools`、`skills`、`instruction`、`depends_on` 等）。  
  - 主任务还可含 `thread_id`、`execution_authorized`、进度与状态等。

实现参考：`backend/packages/harness/deerflow/collab/storage.py` 中 `ProjectStorage`。

### 2. 任务记忆（执行过程事实、摘要、进度等）

- **目录**（默认）：`{base_dir}/.deer-flow/task_memory/`  
- **路径规则**：`task_memory/{project_id}/agents/{agent_id}/{task_id}.json`  
  - 主任务与子任务各自可有记忆文件；`agent_id` 为记忆键的一部分（可与 `assigned_to` 等相关）。  

实现参考：`TaskMemoryStorage`、`load_task_memory_for_task_id`。

### 3. 线程级协作状态（可选）

- 线程目录下 `collab_state.json` 等（如 `bound_task_id`），用于会话与主任务绑定。  
- 与 Gateway **`/api/collab/threads/{thread_id}`** 读写配合。

### 4. HTTP 查询入口（前端不读本地 JSON 时的来源）

- 任务与子任务：`GET /api/tasks/{task_id}` 等（Gateway `tasks` 路由）。  
- 任务记忆：`GET /api/task-memory/...`（以当前路由实现为准）。

## 三、闭环条件（从「能跑」到「用户可感知」）

| 环节 | 条件 |
|------|------|
| 主智能体写完计划 | LangGraph 正常、`supervisor` 落盘成功 |
| 用户能在页面看到任务与子任务 | DeerPanel 调 Gateway 任务 API；或本地同源代理正确 |
| 进度与事件较实时 | LangGraph 进程配置 `DEERFLOW_GATEWAY_URL` 等以便内部广播；前端对 `project_id` 订阅 **`/api/events/projects/{id}/stream`** |
| 聊天里执行子任务带 profile | 运行时 **`subagent_enabled`**；上下文带 `collab_task_id` / `collab_subtask_id`（或聊天页绑定主任务 ID，与线程协作状态一致） |

## 四、实时聊天页面应呈现什么样子

**定位**：用户以自然语言发起/推进任务；可选进入「复杂任务」协作模式，与主任务绑定。

### 4.1 必备区域与行为

1. **对话区**  
   - 用户消息与助手回复（流式）。  
   - 工具调用若需展示：可折叠展示 `supervisor` / `task` 等名称与摘要（避免刷屏可配置）。

2. **会话与线程**  
   - 当前 `thread_id` 与后端一致；新建/切换会话清晰。

3. **复杂任务协作（与任务闭环衔接）**  
   - **开关**：如「复杂任务」模式（当前实现中有类似能力）。  
   - **主任务 ID**：可输入或从任务列表选择后写入上下文，使 **`task` 工具**能带上协作 ID，从存储读取 `worker_profile`。  
   - **与线程绑定**：能将 `bound_task_id` 同步到服务端协作状态（便于阶段中间件与排查）。

4. **快捷入口（建议）**  
   - 从当前会话「跳转到该主任务的任务详情/监控页」**深链**（`#/tasks/{id}` 或任务详情页），避免用户只在聊天里看不到结构化进度。

5. **状态提示**  
   - 未绑定主任务时提示：执行门闩 / `task` 协作参数可能不完整。  
   - 流式中断、重试、停止与现有行为一致。

### 4.2 视觉与信息优先级

- 首屏以**对话**为主；协作 ID、模式为**辅助条**或侧栏，不喧宾夺主。  
- 移动端：协作条可收起为底部抽屉。

## 五、智能体任务进度监控页面应呈现什么样子

**定位**：用户离开聊天后，仍能一眼看到**主任务状态、子任务进度、谁（模板）在执行、技能/工具约束、记忆与事件**。可与「任务列表」「任务详情」合并或分栏，但信息应覆盖下列块。

### 5.1 主任务总览（顶部）

- 名称、描述、状态（规划中/执行中/已完成等）、**整体进度百分比**。  
- **主任务 ID**、**项目 ID**（便于对接 SSE、API）。  
- **绑定线程**（若有）：`thread_id` 或链接回聊天会话。  
- 操作：**开始执行**（对应授权 `start_execution`）、**刷新**、返回项目/任务列表。

### 5.2 子任务时间线或看板（核心）

每个子任务卡片/行建议包含：

| 信息 | 说明 |
|------|------|
| 名称与描述 | 与 `create_subtask` 一致 |
| 子任务 ID | 可复制 |
| 状态 | pending / executing / completed / failed 等 |
| 进度条 | 0–100% |
| **分配模板** | `assigned_to`（即选用的子智能体类型） |
| **worker_profile 摘要** | `base_subagent`、工具列表、技能列表、依赖子任务 ID；可展开全文 |
| 依赖 | `depends_on` 展示为子任务 ID 或名称映射 |

**交互**：点击子任务展开详情（与当前 `task-detail` 等页面对齐）；若已有「配置」徽章，可表示存在 `worker_profile`。

### 5.3 实时更新（SSE）

- 订阅 **`/api/events/projects/{project_id}/stream`**。  
- 收到 `task:progress`、`task:completed`、`task_memory:updated` 等时，**局部刷新**对应子任务或记忆区，无需整页重载。

### 5.4 任务记忆与事实（侧栏或独立 Tab）

- 主任务聚合记忆与子任务记忆（与后端 `task_memory` 聚合策略一致）。  
- 支持按关键词过滤（本地过滤即可）。  
- 展示 `output_summary`、`facts` 列表。

### 5.5 Supervisor / 执行者状态（可选增强）

- **Supervisor 面板**：简短展示最近协调动作或日志摘要（非必须）。  
- **执行者状态**：当前是否有子智能体 `task` 在跑（若前端能拿到流式事件，可显示「运行中」）。

### 5.6 与聊天页的联动

- 从监控页「用当前主任务继续对话」：打开聊天并预填 **主任务 ID**、打开协作模式。  
- 从聊天页「查看任务进度」：跳到本监控/详情页。

## 六、与现有 DeerPanel 的对应关系（便于落地）

- **实时聊天**：`deerpanel/src/pages/chat.js`（含复杂任务开关、主任务 ID、`collab` 上下文等）。  
- **任务列表/详情**：`tasks.js`、`task-detail.js`；详情中含子任务列表、记忆、SSE（`collab-events.js`）。
