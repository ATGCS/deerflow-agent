# 多智能体协作底层数据结构分析

## 📋 概述

本文档详细分析实时对话多智能体协作系统中的所有底层数据结构、存储关系和数据流转。

---

## 🏗️ 核心数据实体

### 实体关系图 (ERD)

```
┌─────────────────┐
│    Project      │ 项目
│  (协作容器)      │
└────────┬────────┘
         │ 1:N
         ▼
┌─────────────────┐
│     Task        │ 任务
│  (主任务/子任务)  │
└────────┬────────┘
         │ N:1
         ▼
┌─────────────────┐
│  WorkerProfile  │ 工作者配置
│  (智能体约束)     │
└────────┬────────┘
         │ 1:1
         ▼
┌─────────────────┐
│ Agent Config    │ 智能体配置
│  (文件系统)       │
└────────┬────────┘
         │ 1:N
         ▼
┌─────────────────┐
│   TaskMemory    │ 任务记忆
│  (执行过程记录)   │
└────────┬────────┘
         │ 1:N
         ▼
┌─────────────────┐
│    TaskFact     │ 事实
│  (提取的关键点)   │
└─────────────────┘

独立实体：
┌─────────────────┐
│ ThreadCollabState│ 线程协作状态
│  (门闩机制)       │
└─────────────────┘

┌─────────────────┐
│  AgentRuntime   │ 智能体运行时
│  (实时状态)       │
└─────────────────┘
```

## 📦 1. Project（项目）

**作用**：协作任务的顶级容器，代表一个完整的用户请求

### 数据结构

```python
class Project(BaseModel):
    id: str                    # 唯一标识符 (UUID 前 8 位)
    name: str                  # 项目名称
    description: str           # 项目描述
    tasks: list[Task]          # 包含的所有任务
    status: ProjectStatus      # 项目状态
    supervisor_session_id: str | None  # Supervisor 会话 ID
    created_at: str            # 创建时间 (ISO8601 Z)
    updated_at: str            # 最后更新时间
```

### 项目状态机

```python
class ProjectStatus(str, Enum):
    PENDING = "pending"       # 等待处理
    PLANNING = "planning"     # 规划中
    EXECUTING = "executing"   # 执行中
    PAUSED = "paused"         # 已暂停
    COMPLETED = "completed"   # 已完成
    FAILED = "failed"         # 失败
    CANCELLED = "cancelled"   # 已取消
```

### 存储位置

```
{base_dir}/projects/{project_id}.json
```

### 示例

```json
{
  "id": "a1b2c3d4",
  "name": "公司官网重构",
  "description": "重构公司官网，包括前端、后端和测试",
  "status": "executing",
  "supervisor_session_id": "sess_001",
  "created_at": "2026-04-04T10:00:00Z",
  "updated_at": "2026-04-04T10:30:00Z",
  "tasks": [
    {
      "id": "task_001",
      "name": "前端开发",
      ...
    },
    {
      "id": "task_002",
      "name": "后端开发",
      ...
    }
  ]
}
```

## 📝 2. Task（任务）

**作用**：可分配给智能体的工作单元，分为主任务（Main Task）和子任务（Subtask）

### 数据结构

```python
class Task(BaseModel):
    id: str                           # 唯一标识符
    name: str                         # 任务名称
    description: str                  # 任务描述
    status: TaskStatus                # 任务状态
    parent_id: str | None             # 父任务 ID（子任务时不为空）
    dependencies: list[str]           # 依赖的任务 ID 列表
    assigned_to: str | None           # 分配给的智能体 ID
    result: Any                       # 执行结果
    error: str | None                 # 错误信息
    created_at: str                   # 创建时间
    started_at: str | None            # 开始时间
    completed_at: str | None          # 完成时间
    progress: int                     # 进度 (0-100)
    execution_authorized: bool        # 门闩 2：是否授权执行
    thread_id: str | None             # 绑定的 LangGraph 线程 ID
    authorized_at: str | None         # 授权时间
    authorized_by: str | None         # 授权方：user | lead | system
```

### 任务状态机

```python
class TaskStatus(str, Enum):
    PENDING = "pending"         # 等待处理
    PLANNING = "planning"       # 规划中（创建子任务）
    PLANNED = "planned"         # 规划完成，等待授权
    EXECUTING = "executing"     # 执行中
    PAUSED = "paused"           # 已暂停
    COMPLETED = "completed"     # 已完成
    FAILED = "failed"           # 失败
    CANCELLED = "cancelled"     # 已取消
```

### 主任务 vs 子任务

| 特性 | 主任务 (Main Task) | 子任务 (Subtask) |
|------|-------------------|-----------------|
| `parent_id` | `null` | 主任务 ID |
| `dependencies` | `[]` | 可能有依赖 |
| `assigned_to` | Lead Agent | 具体执行智能体 |
| `execution_authorized` | 需要授权 | 继承自主任务 |
| 用途 | 用户请求的入口 | 具体工作单元 |

### 示例

```json
{
  "id": "task_001",
  "name": "前端开发",
  "description": "使用 React + TypeScript 开发前端",
  "status": "executing",
  "parent_id": "main_task_001",
  "dependencies": [],
  "assigned_to": "general-purpose",
  "progress": 75,
  "execution_authorized": true,
  "thread_id": "thread_abc123",
  "authorized_at": "2026-04-04T10:15:00Z",
  "authorized_by": "user"
}
```

## 🤖 3. WorkerProfile（工作者配置）

**作用**：定义子任务的执行约束，包括智能体类型、工具白名单、技能等

### 数据结构

```python
class WorkerProfile(BaseModel):
    base_subagent: str | None      # 子智能体模板名（从文件系统加载）
    tools: list[str] | None        # 工具白名单
    skills: list[str] | None       # 技能白名单
    instruction: str | None        # 附加系统指令
    depends_on: list[str] | None   # 依赖的子任务 ID 列表
```

### 字段说明

| 字段 | 类型 | 必填 | 说明 | 示例 |
|------|------|------|------|------|
| `base_subagent` | string | ❌ | 子智能体配置名 | `"general-purpose"`, `"bash"` |
| `tools` | string[] | ❌ | 工具白名单 | `["read_file", "write_file", "bash"]` |
| `skills` | string[] | ❌ | 技能标签 | `["react", "typescript"]` |
| `instruction` | string | ❌ | 特殊指令 | "你是一位前端专家..." |
| `depends_on` | string[] | ❌ | 依赖关系 | `["subtask_001"]` |

### 存储位置

WorkerProfile **不独立存储**，作为子任务的一部分存储在 Task 中：

```json
{
  "id": "subtask_001",
  "name": "前端开发",
  "worker_profile": {
    "base_subagent": "general-purpose",
    "tools": ["read_file", "write_file", "bash"],
    "skills": ["react", "typescript"],
    "instruction": "你是一位前端开发专家...",
    "depends_on": []
  }
}
```

### 传递流程

```
Lead Agent 规划
  │
  ├─► supervisor(create_subtask, worker_profile={...})
  │     └─► 存储到 Task.worker_profile
  │
  ▼
Lead Agent 执行
  │
  ├─► task(collab_subtask_id="sub_001")
  │     └─► task_tool 读取 Task.worker_profile
  │           └─► 提取配置参数
  │                 ├─► subagent_type
  │                 ├─► allowed_tool_names
  │                 ├─► allowed_skill_names
  │                 └─► worker_instructions
  │
  ▼
SubagentExecutor 执行
  └─► 应用配置执行任务
```

## 🧠 4. TaskMemory（任务记忆）

**作用**：记录任务执行过程中的记忆、事实、进度等

### 数据结构

```python
class TaskMemory(BaseModel):
    task_id: str                    # 任务 ID
    agent_id: str                   # 智能体 ID
    project_id: str                 # 项目 ID
    status: TaskStatus              # 任务状态
    facts: list[TaskFact]           # 提取的事实列表
    output_summary: str             # 输出摘要
    current_step: str               # 当前步骤描述
    progress: int                   # 进度 (0-100)
    created_at: str                 # 创建时间
    updated_at: str                 # 最后更新时间
    completed_at: str | None        # 完成时间
```

### 存储位置

```
{base_dir}/task_memory/{task_id}.json
```

### 与 Task 的区别

| 特性 | Task | TaskMemory |
|------|------|------------|
| **用途** | 任务定义和状态 | 执行过程记录 |
| **存储** | `projects/{project_id}.json` | `task_memory/{task_id}.json` |
| **内容** | 结构信息（ID、状态、进度） | 语义信息（事实、摘要、步骤） |
| **更新频率** | 低（状态变更时） | 高（执行过程中持续更新） |
| **生命周期** | 随项目删除 | 可独立保留 |

### 示例

```json
{
  "task_id": "subtask_001",
  "agent_id": "general-purpose",
  "project_id": "a1b2c3d4",
  "status": "executing",
  "facts": [
    {
        "id": "fact_001",
        "content": "项目使用 React 18 + TypeScript",
        "category": "finding",
        "confidence": 0.95
    },
    {
        "id": "fact_002",
        "content": "使用 Tailwind CSS 进行样式设计",
        "category": "decision",
        "confidence": 1.0
    }
  ],
  "output_summary": "已完成首页、关于我们、产品展示、联系我们 4 个页面的开发",
  "current_step": "正在添加路由和导航组件",
  "progress": 75,
  "created_at": "2026-04-04T10:15:00Z",
  "updated_at": "2026-04-04T10:45:00Z",
  "completed_at": null
}
```

## 📌 5. TaskFact（任务事实）

**作用**：从任务执行中提取的关键事实、发现、决策等

### 数据结构

```python
class TaskFact(BaseModel):
    id: str                          # 唯一标识符
    content: str                     # 事实内容
    category: str                    # 分类：finding, decision, data, conclusion
    confidence: float                # 置信度 (0-1)
    source_message: str | None       # 来源消息 ID
```

### 分类说明

| 分类 | 说明 | 示例 |
|------|------|------|
| `finding` | 发现的事实 | "项目使用 React 18" |
| `decision` | 做出的决策 | "选择 Tailwind CSS" |
| `data` | 关键数据 | "API 响应时间 < 200ms" |
| `conclusion` | 得出的结论 | "代码质量良好" |

### 存储位置

TaskFact **不独立存储**，作为 TaskMemory 的一部分：

```json
{
  "task_id": "subtask_001",
  "facts": [
    {
      "id": "fact_001",
      "content": "项目使用 React 18 + TypeScript",
      "category": "finding",
      "confidence": 0.95,
      "source_message": "msg_abc123"
    }
  ]
}
```

## 🚪 6. ThreadCollabState（线程协作状态）

**作用**：管理每个线程的协作阶段和门闩状态

### 数据结构

```python
class ThreadCollabState(BaseModel):
    collab_phase: CollabPhase       # 协作阶段
    bound_task_id: str | None       # 绑定的主任务 ID
    bound_project_id: str | None    # 绑定的项目 ID
    updated_at: str                 # 最后更新时间
```

### 协作阶段机

```python
class CollabPhase(str, Enum):
    IDLE = "idle"                   # 空闲
    REQ_CONFIRM = "req_confirm"     # 需求确认（门闩 1）
    PLANNING = "planning"           # 规划中
    PLAN_READY = "plan_ready"       # 规划完成
    AWAITING_EXEC = "awaiting_exec" # 等待执行授权（门闩 2）
    EXECUTING = "executing"         # 执行中
    DONE = "done"                   # 完成
```

### 存储位置

```
{base_dir}/threads/{thread_id}/collab_state.json
```

**注意**：此文件随线程删除而删除（临时状态）

### 门闩机制

| 门闩 | 阶段 | 条件 | 作用 |
|------|------|------|------|
| **门闩 1** | `REQ_CONFIRM` | 用户明确确认 | 允许进入规划阶段 |
| **门闩 2** | `AWAITING_EXEC` | `execution_authorized=true` | 允许调用 `task` 工具 |

### 示例

```json
{
  "collab_phase": "executing",
  "bound_task_id": "main_task_001",
  "bound_project_id": "a1b2c3d4",
  "updated_at": "2026-04-04T10:30:00Z"
}
```

## 🏃 7. AgentRuntime（智能体运行时）

**作用**：跟踪智能体的实时状态

```python
class AgentRuntime(BaseModel):
```

## 📂 8. Agent Config（智能体配置）

**作用**：定义智能体的能力、工具、系统提示等

### 数据结构

```python
class AgentConfig(BaseModel):
    name: str                           # 智能体名称
    description: str                    # 描述
    agent_type: str                     # custom | subagent | acp
    model: str | None                   # 使用的模型
    tool_groups: list[str] | None       # 工具组
    system_prompt: str | None           # 系统提示（subagent）

    disallowed_tools: list[str] | None  # 工具黑名单
    max_turns: int                      # 最大轮次
    timeout_seconds: int                # 超时时间
    # ... ACP 特有字段
```

### 存储位置

```
{base_dir}/agents/{agent_name}/config.yaml
```

### 示例

```yaml
name: general-purpose
description: 通用智能体，用于复杂多步骤任务
agent_type: subagent
system_prompt: |
  你是一位通用智能体，负责完成复杂任务...
tools:
  - read_file
  - write_file
  - bash
  - web_search
disallowed_tools:
  - task
  - ask_clarification
max_turns: 50
timeout_seconds: 900
```

## 🧩 9. Agent Memory（智能体记忆）

**作用**：智能体的长期记忆，跨任务持久化

### 数据结构

```json
{
  "facts": [
    {
      "text": "用户偏好使用 TypeScript",
      "confidence": 0.95,
      "created_at": 1234567890
    }
  ],
  "context": {
    "user_preferences": {...},
    "domain_knowledge": {...}
  }
}
```

### 存储位置

```
{base_dir}/agents/{agent_name}/memory.json
```

### 与 TaskMemory 的区别

| 特性 | Agent Memory | TaskMemory |
|------|--------------|------------|
| **范围** | 智能体级别（跨任务） | 任务级别（单次执行） |
| **内容** | 长期事实、偏好、知识 | 执行过程、临时事实 |
| **生命周期** | 永久（手动清理） | 随任务结束可清理 |
| **更新频率** | 低（提取重要事实时） | 高（执行过程中） |

## 🔄 数据流转示例

### 场景：用户请求"重构公司官网"

#### 步骤 1：需求确认（门闩 1）

```
用户输入 → Lead Agent 分析
  │
  ├─► ThreadCollabState.collab_phase = "req_confirm"
  │
  └─► ask_clarification("请确认具体需求...")
```

#### 步骤 2：规划

```
Lead Agent 规划
  │
  ├─► supervisor(list_agents, type="subagent")
  │     └─► 读取 {base_dir}/agents/ 目录
  │
  ├─► supervisor(create_task, name="公司官网重构")
```

## 📊 关键设计要点

### 1. 数据分离原则

- **Project/Task**：结构信息（ID、状态、依赖）
- **TaskMemory**：语义信息（事实、摘要、进度）
- **AgentMemory**：长期知识（跨任务持久化）
- **ThreadCollabState**：临时状态（随线程删除）

### 2. 门闩机制

- **门闩 1**（需求确认）：`ThreadCollabState.collab_phase` 控制
- **门闩 2**（执行授权）：`Task.execution_authorized` 字段控制

### 3. WorkerProfile 传递

```
Lead Agent 规划 → 写入 Task.worker_profile
  ↓
task 工具读取 → 提取配置参数
  ↓
SubagentExecutor → 应用约束执行
```

### 4. 记忆提取

```
Subagent 执行 → 输出结果
  ↓
LLM 提取 → TaskFact[]
  ↓
持久化 → TaskMemory.facts
  ↓
（可选）→ AgentMemory.facts（重要长期事实）
```

### 5. 生命周期管理

| 数据 | 创建时机 | 删除时机 | 持久化 |
|------|---------|---------|--------|
| Project | 规划阶段 | 手动删除 | ✅ 永久 |
| Task | 规划阶段 | 随项目删除 | ✅ 永久 |
| TaskMemory | 执行阶段 | 可清理 | ⚠️ 可配置 |
| ThreadCollabState | 线程创建 | 线程删除 | ❌ 临时 |
| AgentRuntime | 执行开始 | 服务重启 | ❌ 临时 |
| AgentMemory | 首次执行 | 手动删除 | ✅ 永久 |
| AgentConfig | 创建智能体 | 手动删除 | ✅ 永久 |
