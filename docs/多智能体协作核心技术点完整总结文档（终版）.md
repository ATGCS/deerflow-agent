# 多智能体协作核心技术点完整总结文档（终版）

> 本文档在保留「多智能体协作」通用原则的同时，**主智能体与子智能体的交互逻辑**已与当前 **DeerFlow（deerflaw 仓库）实现**对齐；若与早期「独立调度器 + DB 唤醒主 LLM」的设想不一致，以本节「二～七」为准。

---

## 一、核心角色定义（DeerFlow 映射）

1. **主智能体（Lead / 全能大脑）**  
   运行在 **LangGraph** 会话中，负责需求对齐、规划、调用 **`supervisor`** 管理协作任务、在适当时机调用 **`task`** 委派子智能体，并汇总对用户的回复。主智能体按 **用户消息 / 工具结果** 触发多轮推理，不是「只在调度器事件下稀疏唤醒」的独立进程模型。

2. **子智能体（Subagent）**  
   由 **`task` 工具** 触发的执行单元：内部使用 **`SubagentExecutor`** + LangChain **`create_agent`**，在 **线程池** 中异步执行，通过 **`astream`** 产生多轮 AI 消息；与主智能体**共用同一套工具实现**（经白名单过滤）、**共用同一线程的沙箱路径**（`thread_id` → `/mnt/user-data/...`）。

3. **「调度」在实现中的含义**  
   并非单独常驻的「任务调度器微服务」。实际是：**LangGraph 运行时** + **`task_tool` 内轮询与 `get_stream_writer` 事件** + **Gateway/SSE 推送** + **协作侧项目 JSON / `collab_state.json`** 等组合完成并发子任务与前端感知。

4. **状态存储模块**  
   默认以 **`ProjectStorage`（项目 JSON）**、**`collab_state.json`（按 thread）**、**任务记忆** 等为主；可扩展 SSE/事件网关。与「必须以 DB/Redis 为中心」的架构无强制绑定，但长任务仍建议保证**磁盘状态与前端解耦**（见第九、十节原则）。

5. **前端交互层**  
   通过 **SSE/WebSocket**（如 Vite 代理至 Gateway）订阅同一条对话流；子智能体过程事件以 **`task_started` / `task_running` / `task_completed`** 等形式汇入**同一次 LangGraph 流**，由前端解析展示。

---

## 二、主智能体调用子智能体的方式（本仓库实现）

### 1. 主路径：`task` 工具

- 主智能体在工具调用中执行 **`task(description, prompt, subagent_type, …, collab_task_id?, collab_subtask_id?)`**。  
- **`task_tool`** 内部创建 **`SubagentExecutor`**：从父运行时传入 **`sandbox` / `thread_data` / `thread_id`**，用 **`get_available_tools`** 取全局工具表，再按子智能体配置与 **`worker_profile`** 做交集过滤。  
- 子智能体使用与主侧同源的 **`create_chat_model`**、**`create_agent`**，并复用 **`build_subagent_runtime_middlewares`**（与主智能体 runtime 中间件体系对齐的子集）。

### 2. 协作路径：`supervisor(start_execution)` 自动委派

- **`supervisor(action="start_execution", …)`** 在完成授权并将协作阶段推进为 **`executing`** 之后，会按 **`subtask_ids`**（若未传则自动选择「已分配且未终态」的子任务）**并行调用上述 `task` 逻辑**（`asyncio.gather`），等价于由服务端替主智能体批量发起子智能体执行。  
- 该调用会 **阻塞至本批子任务对应的 `task` 全部结束**（成功/失败以返回字符串与 `delegatedSubtasks` 为准），主智能体在同一轮工具返回中即可拿到汇总结果。  
- **HTTP `authorize-execution`** 仅做授权与写 `collab_state`，**不**自动起子智能体（无 LangGraph `runtime` 时无法跑 `task`）。

### 3. 与普通工具调用的区别（仍成立）

| 维度 | 子智能体（`task`） | 普通工具（如 `read_file`） |
|------|-------------------|---------------------------|
| 能力 | 多轮推理 + 多工具组合 | 单次确定性调用 |
| 流式 | 子智能体内部 `astream`，经 `task_running` 等事件透出 | 通常一次性返回 |
| 上下文 | 独立子会话，但共享线程沙箱与父级模型/工具源 | 无独立会话 |
| 主智能体 | 等待 `task` 返回字符串结果后再继续推理 | 直接消费工具输出 |

## 三、子智能体流式返回与主智能体的关系（本仓库实现）

1. **子智能体内部**  
   `SubagentExecutor` 对子 Agent **`agent.astream(...)`**，逐步产生 AI 消息，写入内存中的 **`SubagentResult.ai_messages`**。

2. **如何进入主会话流**  
   **`task_tool`** 在异步循环中 **轮询**（默认约每 5 秒）后台任务状态；每当 `ai_messages` 增长，通过 **`get_stream_writer()`** 向 **当前 LangGraph 运行上下文** 写入 **`task_running`**（含单条 message 等），与 **`task_started` / `task_completed` / `task_failed`** 同一套流式协议。

3. **主智能体 LLM 何时「看到」子智能体结果**  
   - **过程**：不通过「把子 token 拼进主 LLM 上下文」实现；前端与同流订阅者通过 **SSE 事件** 看过程。  
   - **结果**：当子智能体跑完，**`task` 工具调用返回** 字符串（如 `Task Succeeded. Result: ...`），该返回值作为 **工具消息进入主智能体下一轮推理**。即：**主 LLM 在 `task` 返回这一拍拿到终态摘要/结果**，而非独立调度服务在任意时刻打断主图。

4. **与旧文档「调度器缓存全文再唤醒主 LLM」的差异**  
   DeerFlow **没有**单独模块按「关键字触发主 LLM 再推理」实现子流消费；主续写由 **LangGraph 正常工具调用结束 → 下一模型步** 驱动。

---

## 四、多子智能体并发（本仓库实现）

1. **同一轮主智能体响应内**  
   可发起多个 **`task` 并行工具调用**（受每轮 **`task` 数量上限** 等策略约束，见 Lead 提示词中的并发说明）。

2. **`start_execution` 内**  
   对多个子任务使用 **`asyncio.gather`** 并行 **`task_tool`**，每个子任务仍由 **`SubagentExecutor`** 的线程池执行；**`tool_call_id`** 区分不同子任务，流式事件里带 **`task_id`** 供前端区分。

3. **防串扰**  
   以 **`tool_call_id` / 内部 background task_id** 及协作 **`collab_subtask_id`** 区分并行子任务；共享 **`thread_id`** 时沙箱路径一致，需注意文件写入路径约定避免互相覆盖。

---

## 五、主智能体对子智能体「长过程」的感知（修正版）

1. **过程感知**  
   主智能体 **LLM 不逐 token 读子智能体**；过程主要通过 **流式事件** 给 **前端/监控**。主 LLM 主要在 **`task` 返回后** 读结构化工具输出。

2. **若需在中间插入主智能体决策**  
   需产品设计为：**缩短单次 `task` 粒度**、或 **多轮对话中由主智能体多次调用 `task`**，而不是依赖文档旧版中的「全局调度器按流触发主 LLM」。

---

## 六、主智能体推理触发时机（修正版）

在 DeerFlow 默认链路中，主智能体推理由 **LangGraph 状态机** 驱动，典型包括：

1. 用户发送新消息  
2. **任一工具返回**（含 **`task` 成功/失败/超时** 的字符串）  
3. **`supervisor` / 其它工具** 返回 JSON 后进入下一轮  
4. 图配置的中断、人工确认等扩展点（若有）

**不再强调**「仅由后台调度器在子流关键字处稀疏唤醒主 LLM」作为本仓库的核心机制。

---

## 七、线程 / 协程模型（修正版）

1. **子智能体执行**  
   **`execute_async`** 将子 Agent 跑在 **线程池**；**`task_tool`** 协程在 **同一条 LangGraph 异步链路** 上 **轮询** 完成态并 **`writer` 推送事件**。  

2. **与主推理的关系**  
   在 **`task` 未返回前**，当前主智能体这一轮 **工具调用未结束**；**返回后** 主模型才基于工具消息继续。这与「接收协程与主推理完全分离、由第三方调度器提交推理任务」的模型不同。

---

## 八、主智能体上下文控制（原则仍适用）

1. 子智能体 **完整思维链** 不默认整段注入主上下文；主侧主要看到 **`task` 返回的摘要字符串**（及可选记忆写入）。  
2. 协作 **任务事实、进度** 可通过 **任务记忆 / 项目 JSON** 持久化，供后续轮次 **`get_task_memory` / `get_status`** 使用。  
3. 超长内容仍建议 **截断、落盘、引用路径** 而非无限塞进主提示词。

---

## 九、长任务稳定运行保障（原则 + 实现提示）

1. **前后端解耦**：LangGraph 运行与 HTTP 短连接解耦；刷新页面应依赖 **thread_id / 会话** 重连流（具体以前端实现为准）。  
2. **持久化**：项目与协作状态落盘；子智能体结束后 **`task_tool`** 可写回 **子任务状态 / 记忆** 并 **SSE 广播**（如 `task:completed`）。  
3. **容错**：子智能体失败/超时通过 **`task` 返回值** 与 **`delegatedSubtasks`** 体现，主智能体可决定重试或改派。

---

## 十、前端实时交互（与本仓库事件对齐）

1. 解析流中的 **`task_started`、`task_running`、`task_completed`、`task_failed`** 等，与子任务 **task_id / description** 绑定展示。  
2. **`supervisor`** 工具 JSON 结果可用于任务侧栏；协作 **`GET /api/collab/threads/...`** 可补充阶段与绑定任务。

---

## 十一、核心禁忌与避坑（更新）

1. 不要认为 **`start_execution` HTTP** 与 **`supervisor(start_execution)`** 行为完全一致：后者才会 **自动委派 `task`**。  
2. 不要假设存在 **独立「任务调度器服务」** 作为子流与主 LLM 的唯一桥梁——实现集中在 **LangGraph + `task_tool` + Executor**。  
3. 多子任务并行时务必区分 **`collab_subtask_id` / tool_call_id**，避免状态写错行。  
4. 长任务仍避免绑定 **单次 HTTP 请求生命周期**（原则不变）。  
5. 避免把 **子智能体原始全量日志** 手工拼进主提示词（原则不变）。

---

## 十二、整体流程总结（DeerFlow 对齐版）

1. 用户在聊天窗口发起任务 → **LangGraph** 以 **thread_id** 运行 **Lead**。  
2. Lead 按需调用 **`supervisor`** 创建主任务、子任务、分配、`start_execution`（或手动 **`task`**）。  
3. **`start_execution`**：授权 + **`executing`** + **并行 `task`** → 子智能体在线程池中运行，事件经 **`get_stream_writer`** 进入 **同一条流**。  
4. 或直接 **`task(collab_task_id, collab_subtask_id)`**：单个子智能体跑完以 **字符串** 回到主会话。  
5. 主智能体在 **工具返回后** 继续推理，结合 **`supervisor(get_status)`** 等更新用户可见结论。  
6. 前端通过 **SSE** 展示工具与子任务事件；协作状态以 **`collab_state.json` + 项目存储** 为准。
