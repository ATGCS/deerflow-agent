## 1. 项目概述

### 1.1 项目背景

DeerFlow 是一个功能强大的多智能体系统，现有 Web 前端偏向开发者使用。为降低使用门槛，需要一个**轻量化的桌面客户端**，专注于日常任务执行、人机协作和技能管理。

### 1.2 设计目标

| 目标 | 说明 |
|-----|------|
| **任务可视化** | 从规划到执行的全过程透明可见 |
| **人机协作** | AI 指挥官 + 人类决策者，各取所长 |
| **技能生态** | 完整的技能生命周期管理 |
| **即开即用** | 全局快捷键，无需配置即可使用 |

### 1.3 核心理念

> **不是让 AI 替代人，而是让 AI 成为最佳副手。**

复杂任务由 **Supervisor（监督协调智能体）** 指挥多个执行 Agent，关键决策点征求人类意见，日常执行自动完成。

### 2.3 典型使用场景

#### 场景 A：快速修复 Bug
```
用户: Cmd+Shift+D 唤起桌面版
→ 输入"修复登录失败的问题"
→ Supervisor 启动，询问范围
→ 用户选择"仅检查 auth 模块"
→ Supervisor 分配 agent-diagnoser 分析
→ 发现问题：token 过期处理逻辑错误
→ Supervisor 询问"是否自动修复？"
→ 用户确认 → agent-coder 执行修复
→ 完成后系统通知，点击查看详情
```

#### 场景 B：复杂项目重构
```
用户: 拖拽项目文件夹到桌面版
→ 选择模板"大型项目重构"
→ AI 规划阶段：
   - Supervisor 拆解为 15 个子任务
   - 设置依赖关系
   - 分配不同 Agent
→ 执行监控：
   - 可视化流程图实时更新
   - Supervisor 发现第 7 任务有循环依赖
   - 询问用户选择处理策略
→ 用户选择"保守方案"
→ Supervisor 调整后续计划
→ 继续执行，最终汇总结果
```

#### 场景 C：安装新技能
```
用户: 打开技能市场
→ 搜索"图表生成"
→ 发现 chart_gen 技能，2.3k 下载
→ 一键安装
→ 立即在新项目中使用
→ 输入"把销售数据可视化"
→ 新技能自动被调用
```

### 3.3 与后端/Agent 的接口映射

**目标**: 约定 Desktop 与现有 DeerFlow 后端（Gateway / LangGraph / backend）的边界和数据流，方便直接开工实现。

- **服务与端口**
  - Gateway: `http://localhost:8001`（Desktop 所有 REST 请求统一打到这里）
  - LangGraph Agent Server: `http://localhost:2024`（SSE / WebSocket 事件源）
  - Web 前端: `http://localhost:2026`（仅作参考，不由 Desktop 直接调用）

- **REST API 映射（逻辑语义，对应后端内部实现可调整）**
  - `GET /api/projects`
    - 由 Gateway 聚合当前登录用户“可见的项目”（可映射到现有 research 项目/会话实体）
  - `POST /api/projects/:id/run`
    - 触发对某个项目的执行：Gateway 启动对应 LangGraph flow（如 `lead_agent` 或自定义 flow），并记录一次执行实例
  - `POST /api/projects/:id/plan`
    - 仅调用“规划/研究 Agent”生成 `Task[]` 计划，不立刻执行；计划保存在后端并通过 Desktop 展示/编辑
  - `POST /api/projects/:id/pause` / `POST /api/projects/:id/resume`
    - 映射到 LangGraph 会话的暂停/恢复（具体实现由 Gateway 封装）
  - `POST /api/projects/:id/ask-supervisor`
    - Desktop 将用户的决策（选项、备注）回传，Gateway 转发给 LangGraph 对应 node 的“人类反馈入口”，并在后端追加一条 `SupervisorDecision`

- **技能相关**
  - `GET /api/skills` / `GET /api/skills/:id`
    - Gateway 从当前配置中读取 skills（包括本地 skills 与 MCP skills），统一返回 Desktop 所需的 `Skill` 结构
  - `GET /api/market/skills` / `POST /api/market/install`
    - 如果暂时没有真实“技能市场”服务，可以由 Gateway 提供 mock 数据，字段对齐 `Skill.marketInfo`

- **SSE 事件规范（LangGraph → Desktop，经由 Gateway）**
  - SSE endpoint: `GET /api/projects/:id/stream`
    - Gateway 代理 LangGraph 的事件流，Desktop 不直接关心底层路由
  - Desktop 最小依赖字段（其余字段原样透传，方便调试）：
    - `task_id: string` → 映射到 `Task.id`
    - `status: 'pending' | 'running' | 'paused' | 'completed' | 'failed'`
    - `progress?: number`（0–100，估算进度）
    - `message?: string`（当前阶段的简要说明或日志）
    - `timestamp: string`
    - `event_type: 'task_update' | 'decision_required' | 'log_append' | ...`

- **错误与鉴权**
  - 所有 Desktop 请求携带统一的访问 token（见「12. 鉴权与账户」），由 Gateway 负责校验并决定可见范围/可执行操作
  - 错误返回结构约定：
    - `4xx`: `{ code: string; message: string; details?: any }`
    - `5xx`: 同结构，但 Desktop 以通用错误文案展示，在 dev 模式下可展开 `details` 调试

### 6.3 数据约束与任务粒度建议

**Task 颗粒度**
- 单个 `Task` 的**理想执行时间**控制在 **1–5 分钟** 内：
  - 超长流程（如“大型项目重构”）拆成多条 `Task`，通过 `dependencies` 串联；
  - 便于进度展示、失败重试和决策复盘。
- `Task.description` 建议写成“可回放”的动作描述（如“扫描 src 目录结构并生成报告”），方便后续在决策历史中理解。

**progress 字段语义**
- `progress: number` 统一约定为 **0–100 的估算值**，由后端/LangGraph 根据任务内部阶段自行计算；
- 对有明确子步骤数量的任务，推荐采用离散进度（如 4 个阶段，每完成一个 +25%）；
- Desktop 只展示后端给出的 `progress`，不做额外推断或本地计算。

**SupervisorDecision 与 Task 的关联**
- 推荐在实际实现中，为 `SupervisorDecision` 增加关联字段（可以在 types 中扩展而不破坏现有结构）：

```typescript
interface SupervisorDecision {
  // ...现有字段...
  target?: { type: 'project' | 'task'; id: string };
  relatedTaskIds?: string[];
}
```

- 用途：
  - 在 UI 中按 Task 查看“该任务相关的所有决策”；
  - 在报告中按 Task 分组展示决策更清晰。

**决策时间线与裁剪**
- 为避免 `Project.decisions` 无限增长，后端推荐：
  - 维护“决策时间线”列表，支持按 `project_id` + 时间分页查询；
  - Desktop 默认只取最近 N 条（如 50 条），滚动时再向后端请求更多；
  - 提供“仅看最近 7 天决策”等过滤选项。

### 9.3 数据同步与缓存策略

**权威数据 vs 本地缓存**
- **权威数据源**：所有项目、任务、技能、决策等业务数据的最终存储都在 DeerFlow 后端（数据库）；
- Desktop 端的 Zustand + IndexedDB 主要用于：
  - 缓存列表/详情，减少重复请求；
  - 在短暂网络抖动时维持 UI 状态连续性；
  - 记录纯前端视图状态（筛选条件、展开折叠等）。
- **不支持离线写入**：
  - 网络不可用时，仅允许浏览已缓存数据；
  - 不允许创建/编辑项目和任务，避免与后端状态冲突。

**多窗口/多实例一致性**
- 同一设备多窗口：
  - 每个窗口有独立的 Zustand 内存状态；
  - 通过 SSE 分别订阅项目/任务更新事件；
  - 一切状态以后端推送为准，不做复杂的本地冲突解决。
- 不同设备之间：
  - 所有写操作先写入后端；
  - 其他设备通过轮询或 SSE 收到更新并刷新 UI。

**错误恢复与重连**
- SSE/WebSocket 断开时：
  - 自动重连；
  - 重连成功后，对当前活跃项目重新拉取一次「项目详情 + 任务列表」，确保状态同步；
- 缓存失效策略：
  - TanStack Query 配置适中的 `staleTime`（如 5–30 秒）；
  - 用户点击“刷新”按钮时始终发起真实请求。

## 12. 鉴权与账户

### 12.1 登录方式

**目标**: 让 Desktop 安全地连接到 DeerFlow 实例，同时尽量减少用户手动配置。

- 推荐模式：浏览器登录
  - Desktop 首次启动时，引导用户在系统默认浏览器中打开 Web 登录页（沿用现有登录/绑定流程）；
  - Web 登录完成后，通过自定义 URL Scheme（如 `deerflow-desktop://auth?code=...`）将一次性 `auth_code` 回传 Desktop；
  - Desktop 使用 `auth_code` 向 Gateway 换取长期访问令牌（`access_token` + 可选 `refresh_token`）。

- 备选模式：手动配置访问 Token
  - 在设置页提供“粘贴 Access Token”的方式，方便开发/测试环境；
  - 明确标注安全风险，不建议在生产环境长期使用。

### 12.2 Token 存储与刷新

- 本地存储：
  - macOS: 使用系统 Keychain；
  - Windows: 使用 Credential Manager 或使用 DPAPI 加密后存本地文件；
  - 不在明文配置文件（如 `.env` / `config.json`）中直接保存访问 Token。
- 刷新策略：
  - 如果 Gateway 提供 `refresh_token`：
    - Desktop 在 `access_token` 即将过期前静默刷新；
    - 刷新失败时，引导用户重新登录；
  - 如果只有长期有效的 `access_token`：
    - 由用户在设置页手动更新；
    - 当请求返回 401/403 时，提示“Token 失效，请重新登录或更新 Token”。

### 12.3 权限与可见范围

- Desktop 端只展示当前登录用户有权限访问的资源：
  - `GET /api/projects`：仅返回该用户有查看权限的项目；
  - `GET /api/skills`：可根据角色控制是否展示管理型/危险技能；
  - 高风险写操作由后端做最终权限判断，Desktop 只做交互层提示。
- 对于“高危决策”（如删除核心函数、批量删除文件）：
  - UI 层：Supervisor 决策框中明确高风险标记 + 二次确认；
  - 后端：仅在用户具备相应角色/权限时才实际执行，其他情况返回 403。
